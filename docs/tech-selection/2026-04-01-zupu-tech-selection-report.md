# 技术选型对比报告（标准版）

- 文档日期：2026-04-01
- 输入文档：[需求分析文档](../requirements/2026-04-01-zupu-requirements-analysis.md)
- 适用阶段：MVP（2-4周）

## 1. 背景与目标

本项目目标是在2-4周内交付可用MVP，打通“手写纸质族谱图片导入 -> OCR识别 -> 人工校对 -> 结构化入库 -> 检索与图谱展示 -> 传统样式导出”的完整链路。系统需支持多家族、角色权限、字段级可见、日志审计，并在3000人规模下满足检索/详情查询P95 < 1s。

本次选型重点聚焦：
1. 能否快速落地P0需求。
2. 是否满足低成本与中等运维复杂度（前后端分离 + 自建服务）。
3. 是否便于后续扩展复杂关系与上云演进。

## 2. 需求约束摘要

来自需求文档与澄清结果的关键约束：
1. 周期：MVP 2-4周。
2. 成本：优先零成本或低成本，开源优先。
3. 架构偏好：前后端分离 + 自建服务。
4. 团队：1-2人，全栈为主。
5. 目标能力：多租户（逻辑隔离）、OCR+人工校对、传统可打印导出、日志审计。
6. 非功能：3000人规模查询P95 < 1s；基础隐私保护（鉴权、权限、日志、备份）。

显式假设：
1. 团队可接受引入一种额外语言组件（例如OCR独立Python服务）。
2. 首版不引入复杂关系（过继、多配偶等）与审批流。

## 3. 候选方案说明

### 候选A：TypeScript主栈（推荐）
- 前端：Vue 3 + TypeScript + Vite
- 后端：NestJS（REST API）
- 数据库：PostgreSQL
- 缓存/队列：Redis + BullMQ
- 文件存储：MinIO（自建对象存储）
- OCR：独立Python OCR服务（PaddleOCR）通过HTTP调用
- 导出：后端HTML模板 + Playwright/Puppeteer生成PDF

适配特点：
1. 与“前后端分离”完全一致。
2. 主开发语言统一为TypeScript，降低1-2人团队沟通和维护成本。
3. OCR能力通过独立服务接入，兼顾质量与后续可替换性。

### 候选B：Python主栈
- 前端：Vue 3 + TypeScript + Vite
- 后端：FastAPI
- 数据库：PostgreSQL
- 异步任务：Celery + Redis
- 文件存储：MinIO
- OCR：PaddleOCR（同语言内整合）
- 导出：WeasyPrint/wkhtmltopdf

适配特点：
1. OCR能力整合自然，算法侧扩展空间大。
2. 但前后端语言分裂，1-2人全栈团队效率可能下降。

### 候选C：图数据库优先栈
- 前端：Vue 3 + TypeScript
- 后端：NestJS/FastAPI（二选一）
- 数据：Neo4j（主）+ PostgreSQL（辅）
- OCR与导出：同A或B

适配特点：
1. 关系查询表达力强，适合未来复杂关系。
2. 首版引入双存储与同步复杂度，交付风险高。

## 4. 评估维度与权重

按Skill固定权重执行加权评分（1-5分）：
1. 功能匹配度：40%
2. 开发效率：30%
3. 可维护性：30%

总分公式：

$$
TotalScore(option)=\sum_{i=1}^{n}(Weight_i \times Score_{i,option})

## 6. 风险与不确定性

1. OCR准确率波动风险
- 不确定性：手写体质量差异大。
- 影响：校对成本上升，可能拖慢MVP节奏。
- 应对：先采用“高准确优先”的人工校对闭环；记录字段纠错率，按周评估OCR服务参数与替换策略。

2. PDF传统排版复杂度风险
- 不确定性：传统族谱版式规则可能频繁调整。
- 影响：导出质量与交付时间。
- 应对：先固化基础模板参数（字号、代际层级、分支换页）；将高级排版规则列入P1迭代。

3. 多租户隔离实现风险
- 不确定性：逻辑隔离在权限遗漏时可能产生越权。
- 影响：数据安全风险。
- 应对：后端统一租户拦截、接口鉴权中间件、越权回归测试用例。

4. 小团队运维负担风险
- 不确定性：自建MinIO/Redis/PostgreSQL需要基础运维能力。
- 影响：稳定性与交付效率。
- 应对：优先单机Docker Compose部署，统一监控与备份脚本，减少手工运维动作。

## 7. 推荐结论与适用条件

推荐方案：候选A（TypeScript主栈 + 独立OCR服务）

推荐理由：
1. 在当前约束下（2-4周、1-2人、前后端分离）综合得分最高。
2. 能在保持交付效率的同时覆盖OCR、导出、权限、审计等核心需求。
3. OCR通过独立服务解耦，后续可平滑替换为更高精度或更低成本方案。

次选方案：候选B（Python主栈）

适用条件：
1. 团队Python能力显著强于TypeScript时。
2. 后续计划强化OCR/算法能力并持续迭代时。

暂不建议：候选C（图数据库优先）

适用条件：
1. 当复杂关系（过继、多配偶、关系推理）进入P0/P1且关系查询成为主要瓶颈时再引入。

需求追溯说明（关键结论依据）：
1. 推荐前后端分离：来源于需求约束中的架构偏好。
2. 选择低成本开源组合：来源于预算约束。
3. 保留OCR解耦：来源于需求中的OCR选型待决项与准确率风险。
4. 暂缓图数据库：来源于MVP边界（复杂关系不在首版范围）。

# 族谱数字化系统需求分析文档（v0.2）

- 文档日期：2026-04-02
- 文档类型：Review-Ready PRD（需求评审版）
- 输入来源：v0.1功能测试反馈、v0.1开发总结

## A. Requirement Summary（简版）

**一句话目标：**
在v0.1 MVP基础上，完成角色权限模型重构、族谱可视化功能、UI优化及已知缺陷修复，使系统达到可交付的完整功能状态。

**关键需求（5条）：**
1. 重构角色权限模型，支持应用管理员/族谱创建者/协作者/普通用户四级权限体系
2. 实现族谱可视化页面，支持树状图和吊线图两种展示形式
3. 修复audits页面无法显示操作日志的问题（需集成审计记录到各业务模块）
4. 全站UI现代化优化，提供更美观的视觉设计和交互体验
5. 扩展PDF导出形式，新增吊线图和行传导出模板

**主要约束（3条）：**
1. 周期约束：开发周期为2-3周
2. 技术约束：保持现有技术栈（Vue3 + NestJS + PostgreSQL）
3. 兼容约束：需兼容v0.1已有数据，提供平滑迁移方案

**首版里程碑：**
1. M1（第1周）：角色权限模型重构完成、audits问题修复
2. M2（第2周）：族谱可视化（树状图+吊线图）上线
3. M3（第2-3周）：UI优化、导出扩展、回归测试完成

**关键待决项：**
1. 协作者修改请求的审批流程细节（同步/异步通知机制）
2. 可视化组件库选型（D3.js / ECharts / 自研）
3. 吊线图/行传的传统版式规范确认

## B. Review-Ready PRD（详版）

### 1. 背景与目标

**背景：**
v0.1 MVP已完成核心闭环（上传→OCR→校对→入库→检索→导出），但实际使用中发现以下关键问题：
1. 角色模型过于简单（仅admin/editor/viewer），无法支持多人协作编辑族谱的真实场景
2. 缺少族谱可视化页面，用户无法直观查看家族关系图谱
3. 部分功能存在缺陷（audits日志不显示、UI粗糙）
4. 导出形式单一，不满足传统族谱多样化排版需求

**业务目标：**
1. 建立完善的多级权限体系，支持族谱创建者管理协作团队
2. 提供直观的可视化展示，增强族谱数据的可读性
3. 修复已知缺陷，确保所有P0/P1功能稳定可用
4. 丰富导出形式，满足不同传统版式需求

**用户目标：**
1. 族谱创建者可独立管理自己创建的族谱及协作团队
2. 用户可通过树状图/吊线图直观浏览家族关系
3. 管理员可完整查看系统操作日志
4. 用户可选择多种导出版式

**成功标准（v0.2）：**
1. 新权限模型覆盖测试文档描述的所有角色场景
2. 可视化页面支持3000人规模族谱流畅展示（P95 < 2s）
3. 所有v0.1已知问题全部修复并通过回归测试
4. 导出支持至少3种版式（速查表、吊线图、行传）

### 2. 用户画像与使用场景

**用户画像：**

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| 应用管理员 | 系统运维人员 | 管理全局用户、族谱、系统配置 |
| 族谱创建者 | 创建族谱的家族负责人 | 全权管理自己的族谱，审批协作请求 |
| 族谱协作者 | 被授权的编辑成员 | 协助录入和维护族谱数据 |
| 普通用户 | 注册用户 | 申请查看特定族谱 |

**核心场景：**

| 场景ID | 场景描述 | 主要角色 |
|--------|----------|----------|
| S1 | 用户创建新族谱，自动成为族谱创建者 | 普通用户→族谱创建者 |
| S2 | 用户通过族谱ID搜索并申请成为协作者/查看者 | 普通用户 |
| S3 | 创建者审批协作者/查看者申请 | 族谱创建者 |
| S4 | 协作者提交修改请求，创建者审批 | 族谱协作者、族谱创建者 |
| S5 | 用户通过树状图/吊线图浏览族谱关系 | 所有有权限用户 |
| S6 | 创建者导出族谱为吊线图/行传PDF | 族谱创建者 |
| S7 | 管理员查看系统操作日志 | 应用管理员 |

**高频路径：**
1. 创建族谱 → 邀请协作者 → 协同录入 → 可视化浏览 → 导出分享
2. 搜索族谱 → 申请查看 → 等待审批 → 浏览可视化
3. 登录 → 切换族谱 → 树状图浏览 → 成员详情

#### FR-101（P0）角色权限模型重构

**描述：**
重构现有角色体系，建立四级权限模型：
1. **应用管理员**：管理系统所有族谱、用户、系统设置
2. **族谱创建者**：管理自己创建的族谱及协作团队
3. **族谱协作者**：协助编辑族谱，可管理查看者
4. **普通用户**：注册用户，可申请查看/协作族谱

**约束：**
- 用户创建族谱后自动成为该族谱的创建者
- 权限继承：创建者权限 > 协作者权限 > 查看者权限
- 需兼容v0.1已有角色数据，提供迁移脚本

**验收标准：**
- [ ] 创建者可审批/拒绝协作者申请
- [ ] 创建者可审批/拒绝查看者申请
- [ ] 创建者可撤销协作者/查看者权限
- [ ] 协作者可审批/拒绝查看者申请
- [ ] 协作者可撤销查看者权限
- [ ] 越权操作被拦截并记录日志

---

#### FR-102（P0）协作者修改审批流程

**描述：**
协作者对族谱的修改需经创建者审批后生效。

**约束：**
- 修改请求包含：修改类型、修改内容、修改原因
- 创建者可选择：通过、拒绝、要求修改
- 修改请求状态持久化，支持查询历史

**验收标准：**
- [ ] 协作者提交修改后，状态为"待审批"
- [ ] 创建者可在待审批列表查看所有修改请求
- [ ] 通过审批后，修改自动生效
- [ ] 拒绝审批时需填写拒绝原因
- [ ] 修改请求历史可追溯

---

#### FR-103（P0）族谱公开程度配置

**描述：**
创建者可配置族谱的公开程度：
1. **所有用户可见**：任何注册用户可直接查看
2. **审批后可见**：需申请并通过审批才能查看

**约束：**
- 默认为"审批后可见"
- 协作者自动获得查看权限
- 配置变更即时生效

**验收标准：**
- [ ] 创建者可切换公开程度配置
- [ ] 公开族谱对所有用户显示在搜索结果中
- [ ] 非公开族谱仅对有权限用户可见
- [ ] 协作者始终可查看其协作的族谱

---

#### FR-104（P0）族谱可视化-树状图

**描述：**
提供族谱树状图展示页面，以图形化方式展示家族成员关系。

**约束：**
- 支持3000人规模流畅渲染
- 支持节点展开/收起
- 支持节点点击跳转成员详情
- 支持缩放、拖拽、全屏

**验收标准：**
- [ ] 树状图正确显示父子关系
- [ ] 配偶关系通过并排或连线展示
- [ ] 3000人规模下初次加载P95 < 2s
- [ ] 节点点击可跳转成员详情页
- [ ] 支持通过搜索定位特定成员节点

---

#### FR-105（P0）族谱可视化-吊线图

**描述：**
提供传统族谱吊线图展示页面，按世代垂直排列。

**约束：**
- 遵循传统族谱版式规范
- 同辈成员水平排列
- 支持世代折叠展开
- 支持导出为图片

**验收标准：**
- [ ] 吊线图按世代正确垂直排列
- [ ] 同辈成员按出生顺序水平排列
- [ ] 支持点击成员查看详情
- [ ] 支持导出当前视图为PNG/JPG

#### FR-106（P0）全站UI现代化优化

**描述：**
对全站页面进行现代化UI优化，提供更美观的视觉设计和流畅的交互体验。

#### FR-107（P1）audits页面日志显示修复

**描述：**
修复audits页面无法显示操作日志的问题。需要：
1. 在成员管理模块（创建/更新/删除）集成审计日志记录
2. 在导出模块集成审计日志记录
3. 在关系管理模块集成审计日志记录

**约束：**
- 保持现有日志数据结构不变
- 支持按时间、操作类型、操作者筛选
- 所有业务模块需调用AuditsService.create()

**验收标准：**
- [ ] MembersService集成审计日志记录
- [ ] ExportsService集成审计日志记录
- [ ] audits页面正确显示所有操作日志
- [ ] 支持按时间范围筛选
- [ ] 支持按操作类型筛选
- [ ] 日志列表支持分页

#### FR-108（P1）PDF导出扩展

**描述：**
扩展PDF导出功能，新增吊线图和行传两种版式。

**约束：**
- 吊线图版式：按世代垂直排列，支持多页
- 行传版式：按世代叙述式排版，含成员生平简介
- 保持速查表版式不变

**验收标准：**
- [ ] 导出界面可选择版式类型
- [ ] 吊线图PDF版式符合传统族谱规范
- [ ] 行传PDF包含成员基础信息和备注
- [ ] 导出内容与在线数据一致
- [ ] 支持预览后导出

#### FR-109（P2）族谱搜索与申请

**描述：**
用户可通过族谱ID或名称搜索族谱，并申请成为协作者/查看者。

**约束：**
- 仅显示公开族谱或有权限的族谱
- 申请需选择申请类型（协作者/查看者）
- 申请可附加申请理由

**验收标准：**
- [ ] 支持按族谱ID精确搜索
- [ ] 支持按族谱名称模糊搜索
- [ ] 搜索结果显示公开程度和当前权限状态
- [ ] 可提交协作者/查看者申请
- [ ] 申请提交后显示待审批状态

---

#### FR-110（P2）申请通知机制

**描述：**
当有新申请或申请状态变更时，通知相关用户。

**约束：**
- MVP阶段采用站内通知
- 通知列表可查看历史
- 支持标记已读

**验收标准：**
- [ ] 新申请时创建者/协作者收到通知
- [ ] 申请审批后申请者收到通知
- [ ] 通知列表显示未读数量
- [ ] 点击通知可跳转相关页面

### 5. 非功能需求

**性能：**
1. 树状图/吊线图3000人规模初次加载P95 < 2s
2. 成员检索/详情查询P95 < 1s（延续v0.1目标）
3. 权限校验不增加额外延迟

**可靠性：**
1. 审批流程需事务保证，避免数据不一致
2. 权限变更需即时生效

**安全与隐私：**
1. 所有权限校验后端强执行，不依赖前端
2. 越权操作记录审计日志
3. 敏感操作（权限变更、数据删除）需二次确认

**兼容性：**
1. 可视化页面在Chrome/Firefox/Edge/Safari最新版可用
2. 响应式布局适配≥1280px宽度屏幕

### 8. 风险、依赖、假设、待决项

**风险清单：**

| 风险 | 触发条件 | 应对措施 |
|------|----------|----------|
| 可视化性能不达标 | 3000人渲染超过2s | 分层渲染、虚拟化、限制初始展开层级 |
| 权限迁移数据丢失 | 迁移脚本bug | 备份后执行、分批迁移、可回滚 |
| 版式规范不明确 | 传统版式规则无文档 | 参考现有族谱样本、用户确认 |

**关键依赖：**
1. 可视化组件库选型确认（建议D3.js）
2. 传统吊线图/行传版式规范样本
3. v0.1数据备份与迁移测试环境

**关键假设：**
1. 用户可接受站内通知方式（非即时推送）
2. 协作者修改审批为异步流程（非实时阻塞）
3. 现有数据库结构可扩展支持新需求

**待决项：**
1. 可视化组件库最终选型
2. 吊线图/行传排版规范细则
3. 通知机制是否需扩展到邮件/短信

## C. System Design Handoff（设计交接输入）

**关键业务流程：**
1. 族谱创建与创建者自动授权流程
2. 协作者/查看者申请与审批流程
3. 协作者修改请求与审批流程
4. 树状图/吊线图数据加载与渲染流程
5. 多版式PDF导出流程

**数据实体与关系（需求视角）：**
1. User 1:N AccessRequest（用户提交的申请）
2. Family 1:N AccessRequest（族谱收到的申请）
3. User 1:N EditRequest（协作者的修改请求）
4. Family 1:1 FamilyVisibility（族谱公开配置）
5. User 1:N Notification（用户通知）

**外部依赖与接口约束：**
1. 可视化渲染依赖前端组件库（D3.js/ECharts）
2. PDF导出继续使用现有pdf-lib方案
3. 所有新接口需集成现有JWT鉴权体系

**非功能目标：**
1. 可视化加载P95 < 2s（3000人规模）
2. 权限校验延迟 < 10ms
3. 申请/审批操作需事务保证

**待技术评审决策点：**
1. 可视化组件选型（D3.js vs ECharts vs GoJS）
2. 树状图/吊线图数据结构设计
3. 权限模型数据库表结构调整方案
4. 修改审批的版本控制策略

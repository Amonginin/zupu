# 族谱数字化系统需求分析文档（MVP）

- 文档日期：2026-04-01
- 文档类型：Review-Ready PRD（需求评审版）
- 输入来源：已确认 plan.md 与需求澄清卡片结果

## A. Requirement Summary（简版）

一句话目标：
构建一个面向多家族的Web族谱系统MVP，在2-4周内打通“手写纸质族谱图片导入、OCR识别、人工校对、结构化入库、检索与图谱展示、传统样式导出”的核心闭环。

关键需求（3-5条）：
1. 支持多家族数据管理与角色权限控制（管理员/编辑/访客）。
2. 支持图片导入与OCR识别，并提供逐条人工校对后入库。
3. 支持成员信息与基础关系（父母/子女/配偶）管理。
4. 支持按姓名/辈分检索与成员关系浏览。
5. 支持传统可打印样式导出（优先PDF）。

主要约束（3条）：
1. 周期约束：MVP交付周期为2-4周。
2. 成本约束：优先零成本或低成本方案（开源与低成本服务优先）。
3. 范围约束：首版不包含复杂关系与审批流。

首版里程碑：
1. M1（第1周）：需求冻结、数据模型与权限模型评审通过。
2. M2（第2周）：OCR+人工校对+入库主流程可跑通。
3. M3（第3-4周）：检索、图谱展示、传统导出、日志审计完成并通过验收。

关键待决项：
1. OCR服务选型（通用OCR服务与开源OCR组合策略）。
2. 导出模板风格细则（传统族谱版式规则细化）。
3. 多租户隔离策略在后续上云时的演进路径。

## B. Review-Ready PRD（详版）

### 1. 背景与目标

背景：
当前族谱主要以纸质与手写记录存在，存在查找困难、维护成本高、共享效率低、易损耗丢失等问题。需要先完成数字化结构沉淀，再逐步提升自动化与协作能力。

业务目标：
1. 降低纸质族谱数字化门槛，形成可持续更新的数据资产。
2. 在首版内建立可用的多家族管理能力与基础协作能力。
3. 产出可打印、可传播的传统族谱导出结果。

用户目标：
1. 管理员可统一维护家族数据、权限与可见性规则。
2. 编辑可高效完成录入、校对、修改。
3. 访客在权限允许范围内进行查看与检索。

成功标准（MVP）：
1. 主流程端到端打通且稳定可演示。
2. 3000人规模下，检索/详情查询P95 < 1s。
3. 导出结果满足“可打印传统样式可用”标准。

### 2. 用户画像与使用场景

用户画像：
1. 家族管理员：负责家族数据治理、权限策略、导出与分享。
2. 家族编辑者：负责资料录入、OCR校对、关系修正。
3. 家族访客：负责浏览与查询公开信息。

核心场景：
1. 场景S1：管理员创建家族空间并分配角色。
2. 场景S2：编辑上传手写族谱图片，OCR识别后逐条校对并入库。
3. 场景S3：用户按姓名/辈分搜索成员并查看关系上下游。
4. 场景S4：管理员导出可打印传统族谱PDF用于线下存档。
5. 场景S5：管理员追溯关键操作日志。

高频路径：
上传图片 -> OCR候选 -> 人工校对 -> 入库 -> 搜索定位 -> 关系浏览 -> 导出。

### 3. 范围边界（In/Out）

In Scope（MVP纳入）：
1. 多家族管理（逻辑隔离）。
2. 角色权限：管理员/编辑/访客。
3. 字段级可见性（由家族管理员配置）。
4. 成员档案与基础关系（父母/子女/配偶）。
5. OCR识别与逐条人工校对入库。
6. 检索（姓名/辈分）与图谱浏览。
7. 传统样式导出（PDF优先）。
8. 操作日志与审计追溯。

Out of Scope（MVP不纳入）：
1. 复杂关系（过继、多配偶、养子女等）。
2. 正式审批工作流（仅日志不审批）。
3. 全自动无校对入库。
4. 移动端原生App与小程序。

### 4. 功能需求（编号、优先级、描述、约束、验收）

优先级矩阵（P0/P1/P2）：
- P0：FR-001, FR-002, FR-003, FR-004, FR-005, FR-006, FR-007
- P1：FR-008, FR-009
- P2：FR-010

FR-001（P0）多家族空间管理
- 描述：系统支持多个家族独立管理数据与成员。
- 约束：首版以逻辑隔离为主，禁止跨家族数据可见。
- 验收：任一用户仅能访问其授权家族数据；跨家族访问被拒绝并记录日志。

FR-002（P0）角色权限控制
- 描述：支持管理员、编辑、访客三角色。
- 约束：访客默认只读，管理员可分配角色。
- 验收：角色权限矩阵可执行，越权操作被拦截。

FR-003（P0）字段级可见性配置
- 描述：管理员可配置访客可见字段。
- 约束：配置后需即时生效，后端强校验。
- 验收：同一成员在不同角色视图中字段展示差异符合配置。

FR-004（P0）OCR导入与人工校对
- 描述：支持上传手写图片，生成OCR候选并逐条确认入库。
- 约束：未校对记录不得入正式库。
- 验收：可完成“上传->识别->校对->入库”全流程；失败项有明确错误原因。

FR-005（P0）成员与关系管理
- 描述：维护成员档案与父母/子女/配偶关系。
- 约束：成员采用系统唯一ID，允许同名；关系冲突需提示。
- 验收：可新增、编辑、删除成员与基础关系，且数据一致性可检查。

FR-006（P0）检索与图谱浏览
- 描述：按姓名、辈分检索，支持成员关系上下游浏览。
- 约束：3000人规模下性能可接受。
- 验收：检索结果准确，P95 < 1s；可从成员详情跳转关系浏览。

FR-007（P0）传统样式导出
- 描述：支持导出可打印传统族谱PDF。
- 约束：导出内容与在线数据一致。
- 验收：导出文件可打印且版式可读，抽检字段与线上一致。

FR-008（P1）操作日志与追溯
- 描述：记录新增、修改、删除、导出、权限变更等关键行为。
- 约束：日志不可被普通角色篡改。
- 验收：可按成员、操作者、时间检索日志。

FR-009（P1）导出前预览
- 描述：导出前可预览版式。
- 约束：预览与最终导出差异可控。
- 验收：预览结果与导出结果核心布局一致。

FR-010（P2）别名检索
- 描述：支持别名字段检索。
- 约束：不影响P0性能目标。
- 验收：配置别名后可命中检索。

### 5. 非功能需求

性能：
1. 3000人规模下，成员检索/详情查询P95 < 1s。
2. OCR任务采用异步处理，用户可查看任务状态。

可靠性：
1. 关键流程失败需可重试并保留错误原因。
2. 提供基础备份能力，防止数据意外丢失。

安全与隐私（基础等级）：
1. 账号鉴权与角色权限必需。
2. 字段级可见控制由后端执行。
3. 关键操作写入审计日志。

兼容性：
1. Web端在主流桌面浏览器可用。
2. 手机浏览器具备基础可用性（查看与检索优先）。

### 6. 数据与接口需求（需求层）

核心数据对象（需求视角）：
1. Family：家族空间与配置。
2. Member：成员基础信息（系统唯一ID）。
3. Relationship：成员之间基础关系。
4. SourceDocument：原始图片与元数据。
5. OcrCandidate：识别候选数据与状态。
6. VisibilityPolicy：字段可见策略。
7. AuditLog：关键行为日志。
8. ExportTask：导出任务与结果。

接口边界（需求视角）：
1. 导入接口：上传原始图片并创建OCR任务。
2. 校对接口：提交修正内容并确认入库。
3. 查询接口：成员检索、详情、关系浏览。
4. 管理接口：角色分配、可见策略配置。
5. 导出接口：触发导出、查询任务状态、下载结果。

### 7. 里程碑与交付标准

M1（第1周）：
1. 需求基线、范围边界、验收口径评审通过。
2. 数据对象与权限矩阵确认。

M2（第2周）：
1. OCR+人工校对+入库主链路可演示。
2. 基础成员与关系管理可用。

M3（第3-4周）：
1. 检索与图谱浏览达到首版可用标准。
2. 传统样式导出与日志追溯可用。
3. 完成MVP验收测试并形成发布清单。

交付标准：
1. 所有P0需求具备可测试验收项且验证通过。
2. 非功能指标达到首版门槛。
3. 待决项有明确后续处理计划。

### 8. 风险、依赖、假设、待决项

风险清单（含触发条件与应对）：
1. OCR手写识别误差高。
- 触发：校对修正比例长期过高。
- 应对：优化字段模板、引入词典规则、评估OCR服务切换。

2. 图谱查询性能退化。
- 触发：3000人规模下P95超过1s。
- 应对：查询索引优化、热点缓存、分页与层级限制。

3. 权限配置复杂导致误暴露。
- 触发：角色访问结果与配置不一致。
- 应对：后端统一鉴权拦截、权限回归用例覆盖。

关键依赖：
1. OCR服务可用性与成本控制。
2. 导出排版模板规则的业务确认。
3. 可用于验收的真实样本数据。

关键假设：
1. 首版用户可接受“高准确优先”的人工校对环节。
2. 复杂关系需求可延后到后续版本。
3. 基础隐私等级可满足初期使用场景。

待决项：
1. OCR引擎最终选型。
2. 导出模板细粒度规范（字号、分支换页、世代布局）。
3. 上云前的多租户隔离增强方案。

## C. System Design Handoff（设计交接输入）

关键业务流程：
1. 家族创建与角色授权流程。
2. 图片导入、OCR、人工校对、入库流程。
3. 成员检索与关系浏览流程。
4. 导出与下载流程。
5. 审计日志查询流程。

数据实体与关系（需求视角）：
1. Family 1:N Member。
2. Member N:N Member（通过Relationship表示父母/子女/配偶）。
3. SourceDocument 1:N OcrCandidate。
4. Family 1:N VisibilityPolicy。
5. Family 1:N AuditLog。
6. Family 1:N ExportTask。

外部依赖与接口约束：
1. OCR外部服务需支持中文手写识别输入。
2. 导出能力需支持可打印PDF输出。
3. 所有跨家族数据访问必须被接口层阻断。

非功能目标（SLO/容量/时延）：
1. 容量目标：首版单家族可支撑3000成员规模。
2. 时延目标：检索/详情查询P95 < 1s。
3. 可用性目标：主流程可重试且失败可追踪。

待技术评审决策点：
1. OCR服务与回退策略。
2. 数据存储模型对关系查询性能的影响评估。
3. 导出渲染方案在可读性与性能上的平衡。
4. 多租户隔离策略从MVP到上云阶段的演进。
